# Kubernetes Cluster Configuration

This document describes the steps required to configure a Kubernetes cluster using Ansible and Debian virtual machines.

## 1. Installing Ansible

We will install Ansible on our machine to manage the virtual machines.

- On macOS:

```bash
brew install ansible
ansible --version
```

> Note: Ensure that the Debian servers have Python installed and the SSH service enabled. The Debian 13 image used by Terraform already includes both.

## 2. The Inventory (hosts.ini)

`hosts.ini` is **generated by Terraform** (see `local_file.ansible_inventory` in `../main.tf`), so it always holds the current droplet IPs. Do not edit it by hand: the next `terraform apply` overwrites it.

```ini
[master]
cp01 ansible_host=<cp01-ip> ansible_user=root

[workers]
wk01 ansible_host=<wk01-ip> ansible_user=root
```

## 3. SSH Access

Terraform attaches your public key (`~/.ssh/id_digitalocean_kubeadm.pub`) to both droplets when it creates them, so passwordless access from your machine works out of the box. There is no `ssh-copy-id` step: the images ship with `PasswordAuthentication no`, so it could not authenticate anyway.

Verify it:

```bash
ansible all -m ping
```

If it fails with `Permission denied (publickey)`, your local private key is not the one Terraform uploaded. See section 4.

Access **between** the nodes (master to workers, needed by `post-install.sh`) is a separate key, created and distributed by the playbook itself — see `roles/kubernetes/tasks/ssh_trust.yml`.

## 4. Configuring the ~/.ssh/config File

If you have multiple SSH keys, configure `~/.ssh/config` so the right key is used for each node. Take the IPs from the generated `hosts.ini`:

```bash
Host <cp01-ip> <wk01-ip>
    User root
    IdentityFile ~/.ssh/id_digitalocean_kubeadm
    IdentitiesOnly yes
```

## 5. Basic Ansible Commands

Test the connection with the VMs:

```bash
ansible -i hosts.ini all -m ping
```

Run the playbook:

```bash
ansible-playbook -i hosts.ini kube-play.yml
```

The playbook installs containerd, the Kubernetes binaries and the shell settings, sets up passwordless SSH from the master node to the workers, and copies `post-install.sh` plus `hosts.ini` to `/root/` on the master. **It does not create the cluster** — that is the next step.

## 6. Post-Installation: Create the Cluster

Once the playbook finishes, connect to the master node and run the script:

```bash
ssh root@<master-ip>
cd /root && ./post-install.sh
```

`post-install.sh` runs `kubeadm init`, configures `kubectl` for `root`, installs Calico, and joins every worker listed in `hosts.ini`.

The script is idempotent: it checks the state of each step before acting, so you can re-run it as many times as you need (for example, after adding a new worker to `hosts.ini`).
